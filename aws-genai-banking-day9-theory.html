Module 1: Fine-tuning vs RAG (60 min)

üéØ Fine-tuning vs RAG: Choosing the Right Approach

Understanding when to use each technique

What is RAG (Retrieval-Augmented Generation)?

üìö RAG Overview

  • Retrieval: Search relevant documents from knowledge base
  • Augmentation: Add retrieved context to user query
  • Generation: LLM generates response with context
  • Benefits: Up-to-date info, citations, no retraining

üîß Fine-tuning Overview

  • Training: Retrain model on custom dataset
  • Adaptation: Model learns new behavior/style
  • Deployment: Use fine-tuned model for inference
  • Benefits: Consistent behavior, domain expertise

Architecture Comparison

Detailed Comparison

Aspect Fine-tuning RAG Recommendation
Cost High ($1000s) Low ($10s) RAG for most cases
Time to Deploy Days/weeks Minutes/hours RAG for speed
Data Required 1000s examples Any documents RAG for flexibility
Updates Retrain model Update docs RAG for dynamic data
Accuracy Very high (95%+) High (85-90%) Fine-tuning for critical
Explainability Low (black box) High (citations) RAG for transparency
Maintenance High (retraining) Low (doc updates) RAG for ease

Banking Use Cases

üîß When to Use Fine-tuning

  • Legal Document Generation: Learn specific legal writing style
  • Credit Risk Modeling: Domain-specific risk assessment
  • Fraud Detection: Learn fraud patterns from historical data
  • Regulatory Reporting: Consistent format and terminology

üìö When to Use RAG

  • Banking Policy Q&A: Up-to-date policy information
  • Product Information: Current rates and features
  • Regulatory Compliance: Latest regulations with citations
  • Customer Support: Knowledge base with source attribution
üí°
Best Practice: Start with RAG for most use cases. It's faster, cheaper, and easier to maintain. Use fine-tuning only when you need the model to learn new behavior or achieve very high accuracy for critical tasks.

Chunking Strategies

üìè Fixed Size

  • Size: 500-1000 tokens
  • Overlap: 50-100 tokens
  • Pros: Simple, consistent
  • Cons: May split context

üìÑ Semantic

  • Method: Split by paragraphs
  • Size: Variable
  • Pros: Preserves meaning
  • Cons: Complex logic

üéØ Hybrid

  • Method: Semantic + size limit
  • Size: Max 1000 tokens
  • Pros: Best of both
  • Cons: Most complex

üè¶ SecureBank Example: Banking Policy Knowledge Base

  • Data Source: 500 policy documents in S3 (mortgage, loans, compliance)
  • Chunking: 800 tokens with 100 token overlap = 2,500 chunks
  • Embeddings: Titan Embeddings (1536 dims) = 3.8M vectors
  • Vector Store: OpenSearch Serverless (2 OCUs) = $0.20/hr = $144/month
  • Query Performance: <100ms latency, 95% accuracy
üí°
Best Practice: For banking documents, use hybrid chunking with 800 tokens and 100 token overlap. This preserves context while keeping chunks manageable. Always include metadata (document name, section, date) for better filtering.

Module 3: Bedrock Agents (60 min)

ü§ñ Bedrock Agents & Task Orchestration

Intelligent agents with action groups and knowledge bases

What are Bedrock Agents?

ü§ñ Agent Overview

  • Definition: Autonomous AI that can plan and execute tasks
  • Orchestration: Breaks complex requests into steps
  • Actions: Calls APIs and Lambda functions
  • Knowledge: Retrieves info from knowledge bases

üéØ Key Components

  • Foundation Model: Claude 4.5 for reasoning
  • Action Groups: Lambda functions for tasks
  • Knowledge Bases: RAG for information
  • Instructions: Agent behavior and personality

Action Groups vs Knowledge Bases

‚ö° Action Groups

  • Purpose: Execute tasks and operations
  • Implementation: Lambda functions with OpenAPI schema
  • Examples: Transfer money, check balance, create account
  • When to Use: Need to perform actions or call APIs

üìö Knowledge Bases

  • Purpose: Retrieve information and answer questions
  • Implementation: Vector search with RAG
  • Examples: Policy lookup, FAQ answers, document search
  • When to Use: Need to provide information with citations

Banking Agent Use Cases

üí∞ Loan Application Agent

  • Collect customer information
  • Check credit score (action)
  • Retrieve loan policies (KB)
  • Calculate loan terms (action)
  • Submit application (action)

üè¶ Account Management Agent

  • Check balances (action)
  • Transfer money (action)
  • Update profile (action)
  • Answer FAQs (KB)
  • Order cards (action)

üìä Investment Advisory Agent

  • Analyze portfolio (action)
  • Retrieve market data (action)
  • Get investment policies (KB)
  • Recommend investments (KB)
  • Execute trades (action)
üí°
Best Practice: Combine action groups and knowledge bases in a single agent. Use action groups for operations (transfers, updates) and knowledge bases for information retrieval (policies, FAQs). This creates a powerful agent that can both answer questions and take actions.

LangChain RAG Implementation

1Ô∏è‚É£ Basic RAG with LangChain

from langchain_aws import BedrockLLM, BedrockEmbeddings from langchain_community.vectorstores import OpenSearchVectorSearch from langchain.chains import RetrievalQA # Initialize Bedrock LLM llm = BedrockLLM( model_id="anthropic.claude-sonnet-4-5-20250929-v1:0", region_name="us-east-1" ) # Initialize embeddings embeddings = BedrockEmbeddings( model_id="amazon.titan-embed-text-v1" ) # Connect to OpenSearch vector store vector_store = OpenSearchVectorSearch( opensearch_url="https://your-opensearch-endpoint", index_name="banking-docs", embedding_function=embeddings ) # Create RAG chain qa_chain = RetrievalQA.from_chain_type( llm=llm, retriever=vector_store.as_retriever( search_kwargs={"k": 3} # Top 3 docs ) ) # Query result = qa_chain("What is the mortgage approval process?") print(result)

2Ô∏è‚É£ RAG with Source Citations

from langchain.chains import RetrievalQAWithSourcesChain # Create chain with sources qa_with_sources = RetrievalQAWithSourcesChain.from_chain_type( llm=llm, retriever=vector_store.as_retriever( search_kwargs={"k": 3} ), return_source_documents=True ) # Query with citations result = qa_with_sources( "What are the KYC requirements for corporate accounts?" ) print(f"Answer: {result['answer']}") print(f"Sources: {result['sources']}") # Print source documents for doc in result['source_documents']: print(f"- {doc.metadata['source']}")

3Ô∏è‚É£ Conversational RAG

from langchain.chains import ConversationalRetrievalChain from langchain.memory import ConversationBufferMemory # Add conversation memory memory = ConversationBufferMemory( memory_key="chat_history", return_messages=True, output_key="answer" ) # Create conversational chain conversation_chain = ConversationalRetrievalChain.from_llm( llm=llm, retriever=vector_store.as_retriever(), memory=memory ) # Multi-turn conversation response1 = conversation_chain( "What savings account types do you offer?" ) response2 = conversation_chain( "What are the interest rates?" # Remembers context )

4Ô∏è‚É£ LlamaIndex RAG

from llama_index.llms.bedrock import Bedrock from llama_index.embeddings.bedrock import BedrockEmbedding from llama_index.core import VectorStoreIndex, ServiceContext # Initialize LlamaIndex with Bedrock llm = Bedrock( model="anthropic.claude-sonnet-4-5-20250929-v1:0" ) embed_model = BedrockEmbedding( model="amazon.titan-embed-text-v1" ) # Create service context service_context = ServiceContext.from_defaults( llm=llm, embed_model=embed_model ) # Create index and query index = VectorStoreIndex.from_documents( documents, service_context=service_context ) query_engine = index.as_query_engine() response = query_engine.query( "What is the loan approval process?" )

RAG Best Practices

üéØ Retrieval Optimization

  • Top-K: Start with k=3, adjust based on results
  • Hybrid Search: Combine keyword + vector
  • Metadata Filtering: Filter by date, category
  • Re-ranking: Use cross-encoder for better results

üìù Prompt Engineering

  • Context Window: Keep under 4K tokens
  • Instructions: Clear, specific prompts
  • Citations: Always ask for sources
  • Fallback: Handle no-results gracefully

‚ö° Performance

  • Caching: Cache embeddings and results
  • Batch Processing: Process multiple queries
  • Async: Use async for parallel retrieval
  • Monitoring: Track latency and accuracy

üè¶ SecureBank RAG Implementation

  • Documents: 500 banking policies, 1,000 FAQs, 200 product docs
  • Chunking: 800 tokens with 100 overlap = 8,500 chunks
  • Vector Store: OpenSearch Serverless (4 OCUs) = $0.40/hr = $288/month
  • Embeddings: Titan Embeddings = $0.0001/1K tokens = $8.50/month
  • LLM: Claude Sonnet 4.5 = $0.003/1K input + $0.015/1K output = ~$50/month
  • Total Cost: ~$350/month for 10K queries/month
  • Performance: 200ms latency, 90% accuracy, 95% user satisfaction
üí°
Production Tip: Start with LangChain for rapid prototyping. It has excellent Bedrock integration and built-in chains. For advanced use cases (custom retrievers, complex pipelines), consider LlamaIndex. Both frameworks work seamlessly with Bedrock and OpenSearch.
